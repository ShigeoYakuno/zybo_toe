# TCP/IP Offload Engine (TOE) のメリット — lwIP との比較

## 1. TOE とは

TOE (TCP/IP Offload Engine) は、通常ならソフトウェアスタックが担う **ネットワークプロトコル処理を FPGA の回路 (PL: Programmable Logic) で実行する**アーキテクチャです。  
CPU (ARM など) は AXI4-Lite レジスタに数バイト書くだけで送受信が完結し、イーサネットフレームの組み立て・解析・チェックサム計算などは一切行いません。

このプロジェクトでは ZYBO Z7-20 上に独自 TOE IP コアを実装し、ARP / UDP を FPGA ハードウェアで処理しています。

---

## 2. lwIP との構造比較

```
【lwIP (ソフトウェアスタック)】

  センサ / アプリ
      │
  BSD socket API
      │
  lwIP TCP/UDP 層      ← ARM で実行 (CPU サイクル消費)
      │
  lwIP IP 層           ← チェックサム計算・ARP テーブル管理
      │
  lwIP Ethernet 層     ← フレーム組み立て・DMA 管理
      │
  MAC DMA (GEM など)
      │
  PHY (LAN8720A)
      │
  ネットワーク


【TOE (このプロジェクト)】

  センサ / アプリ
      │
  AXI4-Lite レジスタ書き込み (数命令)   ← ARM が行うのはここだけ
      │
  TOE IP コア (FPGA PL)
  ┌─────────────────────────────┐
  │ ARP エンジン                │  ← ハードウェアで自動処理
  │ UDP/IP 組み立て・解析       │  ← ハードウェアで自動処理
  │ チェックサム計算            │  ← ハードウェアで自動処理
  │ TX/RX 非同期 FIFO           │
  └─────────────────────────────┘
      │
  RMII (LAN8720A)
      │
  ネットワーク
```

## 3. 主なメリット

### 3-1. CPU 負荷ゼロのプロトコル処理

lwIP では ARM が IP ヘッダ生成・UDP チェックサム計算・ARP テーブル検索などを毎パケット実行します。  
TOE ではこれらすべてが PL 回路で完結するため、**ARM の CPU サイクルはアプリケーション処理だけに使えます**。

| 処理 | lwIP | TOE |
|------|------|-----|
| Ethernet フレーム生成 | ARM | PL |
| IP ヘッダ / チェックサム | ARM | PL |
| UDP ヘッダ | ARM | PL |
| ARP 要求・応答 | ARM | PL (自動) |
| ARM の関与 | 全処理 | レジスタ R/W のみ |

### 3-2. 決定論的レイテンシ

lwIP はソフトウェアスタックのため、割り込み遅延・OS スケジューラ・キャッシュミスなどでレイテンシが揺れます。  
TOE の送信パスは **レジスタへの書き込み → send_req パルス → 固定クロック数で送信開始** という純粋なハードウェアシーケンスであり、ジッタがほぼゼロです。

このプロジェクトでは 64 バイト UDP フレームを **50 MHz RMII で約 9.4 µs** で送信しており、CPU の状態に関わらず毎回この時間で完了します。

### 3-3. OS・RTOS 不要

lwIP を裸金属 (Bare-metal) で動かすにも、タイマ割り込みや pbuf メモリ管理などの仕組みが必要です。  
TOE はレジスタポーリングだけで動作するため、**FreeRTOS も割り込みコントローラも不要**。最小構成のベアメタルアプリで完結します。

### 3-4. メモリフットプリントが小さい

| 項目 | lwIP | TOE |
|------|------|-----|
| TCP/IP スタック本体 | 数十〜数百 KB (RAM) | 不要 |
| DMA バッファ | 必要 | PL 内 FIFO |
| ARP テーブル | ARM SRAM | PL 内レジスタ |
| アプリ側バッファ | 数 KB | 64 バイト固定 |

lwIP を動かすには最低でも数十 KB の RAM とヒープが必要ですが、TOE では ARM 側に必要なのは **レジスタアクセス用のわずかな変数のみ**です。

### 3-5. センサ処理とネットワーク処理の並列実行

lwIP では ARM が送受信処理を占有している間、センサ読み取りは待たされます。  
TOE では **FPGA がパケット処理を並行して実行**するため、ARM はセンサ読み取り・演算・表示をしながら、裏でネットワーク送受信が進みます。

```
lwIP:    [センサ] → [ネットワーク処理] → [センサ] → ...
TOE:     [センサ]
                   [PL が TX/RX 処理]  ← 並列
         [センサ] → [センサ] → ...
```

## 4. lwIP が有利な場面

TOE は万能ではなく、以下のケースでは lwIP が現実的な選択です。

| ケース | 理由 |
|--------|------|
| TCP が必要 | 再送・輻輳制御などは FPGA 実装が大規模になる |
| 多数のコネクション | ステート管理を PL に置くとリソースが逼迫する |
| ポータビリティ重視 | lwIP はどの CPU でも動く |
| 開発リソースが少ない | lwIP は枯れたライブラリで学習コストが低い |
| ペイロードが可変長 | このプロジェクトの TOE は固定 64 バイト前提 |

## 5. このプロジェクトでの実装概要

| 項目 | 内容 |
|------|------|
| ターゲット | ZYBO Z7-20 (Zynq-7020) |
| PHY | Waveshare LAN8720A (RMII, PMOD JC) |
| プロトコル | UDP / ARP |
| ペイロード | 固定 64 バイト |
| クロック | 50 MHz (RMII 基準クロック) |
| レジスタ I/F | AXI4-Lite (ベースアドレス 0x40000000) |
| TX FIFO | 2048 バイト (非同期 FIFO, PL 内) |
| RX FIFO | 4096 バイト (非同期 FIFO, PL 内) |
| 送信レイテンシ | ~9.4 µs (64 バイトフレーム送出時間) |
| ARM の送信手順 | TX_DATA に 64 バイト書き込み → CTRL[0] パルス |

ARM 側のコードは AXI レジスタ操作のみで構成されており、`send64` コマンドの実装はわずか数十行です。

## 6. まとめ

| 評価軸 | lwIP | TOE |
|--------|------|-----|
| CPU 負荷 | 高 (プロトコル処理を ARM が担う) | 低 (レジスタ R/W のみ) |
| レイテンシ安定性 | OS/割り込み依存でジッタあり | ハードウェア固定、ジッタなし |
| メモリ使用量 | 数十〜百 KB | 最小限 |
| OS 依存 | タイマ・割り込みが必要 | 不要 |
| リアルタイム性 | 低〜中 | 高 |
| 実装コスト | ライブラリ流用で低 | HDL 設計が必要 |
| TCP 対応 | ◎ | △ (追加設計が必要) |
| 適用場面 | 汎用・多接続・TCP | 組み込み・低レイテンシ・UDP |

TOE は「**決まったフォーマットのデータを高速・低レイテンシ・低 CPU 負荷で送りたい**」組み込みシステムに最適なアーキテクチャです。センサデータの定周期送信、工場ラインの制御信号配信、計測器のストリーミングなど、プロトコルが固定できる用途で真価を発揮します。
